# Task 0.6 — Configure Supabase Row-Level Security

## Overview

Implement Row-Level Security (RLS) policies to ensure users can only access their own data, while admins have full access.

## Prerequisites

---

## RLS Overview

**Row-Level Security** restricts which rows a user can read, insert, update, or delete based on their identity.

For Legacy Scale:
- **Members** can only access their own data (enrollments, progress, notes)
- **Admins** have full access to all data
- **Public data** (published courses, announcements) readable by all authenticated users

### Step 3: Create Helper Function for Admin Check

```sql
-- Helper function to check if user is admin
CREATE OR REPLACE FUNCTION is_admin()
RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM users
    WHERE id = auth.uid()::text
    AND role = 'ADMIN'
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### Step 4: Users Table Policies

```sql
-- Users: Read own profile
CREATE POLICY "Users can read own profile"
ON users FOR SELECT
USING (id = auth.uid()::text);

-- Users: Update own profile
CREATE POLICY "Users can update own profile"
ON users FOR UPDATE
USING (id = auth.uid()::text)
WITH CHECK (id = auth.uid()::text);

-- Admins: Full access to users
CREATE POLICY "Admins have full access to users"
ON users FOR ALL
USING (is_admin());
```

## Important Notes

### Service Role Key Bypasses RLS
When using `SUPABASE_SERVICE_ROLE_KEY`, RLS is bypassed. Use this only for:
- Admin operations
- Webhooks
- Server-side operations that need full access

### User ID Matching
The policies use `auth.uid()::text` to match the user ID from Supabase Auth with the `user_id` column (which is a TEXT/UUID field).

### Admin Function Security
The `is_admin()` function uses `SECURITY DEFINER` which means it runs with the permissions of the function owner (postgres), allowing it to check the users table even when RLS would normally block it.

## Troubleshooting

### "Permission denied for table X"
- Check RLS is enabled on the table
- Check policies exist for the operation
- Verify the user's role matches policy requirements

### "Function is_admin() does not exist"
- Run the CREATE FUNCTION statement first
- Check for syntax errors

### Policies not working
- Make sure you're using the anon key, not service role key
- Check auth.uid() returns the correct user ID
- Test with Supabase SQL Editor using `set request.jwt.claim.sub = 'user-id'`

### "duplicate key value violates unique constraint" on policy creation
- Policy already exists
- Drop it first: `DROP POLICY "policy_name" ON table_name;`
